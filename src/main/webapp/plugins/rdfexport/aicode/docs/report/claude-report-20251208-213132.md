# Bug Fix Report: literal_definitions Not Propagating to Python Parser

**Date:** 2025-12-08
**Branch:** `claude/debug-typescript-plugin-011mrWJio84i9nQ5z2EpL1PY` (based on `feat/mint-literal-knobs`)
**Status:** Fixed

## Executive Summary

Fixed a bug where `literal_definitions` configuration from the TypeScript plugin UI was not being propagated to the Python parser (`parse_drawio_xml`).

**There were TWO bugs:**

1. **Primary Bug (TypeScript):** The `normalizeLiteralEntries` function only checked for snake_case keys (`attr_key`, `attr_value`) from YAML, but ignored camelCase keys (`attrKey`, `attrVal`) from the UI.

2. **Secondary Bug (Python):** The `_apply_parser_overrides` function was missing a handler for `literal_definitions` overrides (though this was less impactful since the TypeScript bug prevented any UI values from reaching Python anyway).

## Problem Description

### Symptoms
- When users configured `literal_definitions` in the parser settings dialog, the settings were ignored by the Python parser
- `literal_definitions` always came as the default from YAML (e.g., `[{attr_key: "style", attr_value: "rounded=1"}]`), regardless of user settings
- When the user explicitly cleared all literal definitions (empty array), the defaults were still used

### Expected Behavior
- If user sets custom `literal_definitions`, Python should receive them
- If user clears all definitions (empty array `[]`), Python should receive an empty array
- If no override is provided (`null`), Python should use defaults from `default.yml`

## Root Cause Analysis

### Bug #1: TypeScript Property Name Mismatch (PRIMARY)

The `normalizeLiteralEntries` function in `rdfexport.ts` was the main culprit.

**Data format from UI (camelCase):**
```typescript
// In normaliseParserSettings, when called with UI data:
literalDefinitions: literalDefEntries.map((entry) => ({
  attrKey: entry.keyInput.value,   // camelCase
  attrVal: entry.valueInput.value, // camelCase
})),
```

**Data format from YAML (snake_case):**
```yaml
literal_definitions:
  - attr_key: style      # snake_case
    attr_value: rounded=1 # snake_case
```

**Broken normalizeLiteralEntries (only checked snake_case):**
```typescript
const attrKey =
  typeof (entry as LiteralParserConfigEntry).attr_key === "string"
    ? (entry as LiteralParserConfigEntry).attr_key  // Only snake_case!
    : "";
```

When UI entries (with `attrKey`) were passed, the check for `attr_key` failed, resulting in empty string `""`, which then caused the entry to be skipped at line 501-503.

**Contrast with metacharacter_substitute:** This worked because both YAML and UI use the same property names (`character`, `replacement`).

### Bug #2: Python Missing Handler (SECONDARY)

The `_apply_parser_overrides` function in `drawio_pipeline.py` had no handler for `literal_definitions`, so even if TypeScript sent them correctly, Python would ignore them.

## Solution

### Fix #1: TypeScript - Support Both Formats

**File:** `aicode/typescript_plugin/src/rdfexport.ts`

Changed `normalizeLiteralEntries` to check both snake_case (from YAML) and camelCase (from UI):

```typescript
function normalizeLiteralEntries(
  entries: unknown,
): LiteralParserSettingsEntry[] | null {
  // ...
  for (const entry of entries) {
    // ...
    // Check both snake_case (from YAML) and camelCase (from UI) formats
    const entryObj = entry as Record<string, unknown>;
    const attrKey =
      typeof entryObj.attr_key === "string"
        ? entryObj.attr_key
        : typeof entryObj.attrKey === "string"
          ? entryObj.attrKey
          : "";

    // ... same pattern for attrVal ...
  }
}
```

### Fix #2: Python - Add Handler for literal_definitions

**File:** `aicode/python_core/pyodide_pipeline/drawio_pipeline.py`

1. Added `_normalise_literal_definitions()` function
2. Added handler in `_apply_parser_overrides()`:

```python
if "literal_definitions" in overrides:
    normalised = _normalise_literal_definitions(
        overrides["literal_definitions"]
    )
    if normalised is not None:
        config["literal_definitions"] = normalised
```

### Test Added

**File:** `aicode/typescript_plugin/tests/rdfexport.test.ts`

Added test `"runDrawioPipeline forwards literal_definitions to parser configuration"` that verifies:
- `null` -> use defaults from YAML
- `[]` -> user explicitly cleared (empty array)
- Custom values -> use those values

## Technical Notes

### Why metacharacter_substitute Worked But literal_definitions Didn't

The key difference is in property naming conventions:

**metacharacter_substitute:**
- YAML: `character`, `replacement`
- UI: `character`, `replacement`
- Same names in both formats ✓

**literal_definitions:**
- YAML: `attr_key`, `attr_value` (snake_case)
- UI: `attrKey`, `attrVal` (camelCase)
- Different names → UI values were silently discarded ✗

### Semantic Difference: `null` vs `[]`

This fix preserves an important semantic distinction:
- `null`: "No preference specified, use defaults from YAML"
- `[]`: "User explicitly cleared all definitions"

This is critical because the Python parser uses `literal_definitions` to determine which cells should be treated as literals vs objects.

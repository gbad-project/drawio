# Bug Fix Report: literal_definitions Not Propagating to Python Parser

**Date:** 2025-12-08
**Branch:** `claude/debug-typescript-plugin-011mrWJio84i9nQ5z2EpL1PY` (based on `feat/mint-literal-knobs`)
**Status:** Fixed

## Executive Summary

Fixed a bug where `literal_definitions` configuration from the TypeScript plugin was not being propagated to the Python parser (`parse_drawio_xml`). The root cause was missing handling for the `literal_definitions` field in the `_apply_parser_overrides` function in `drawio_pipeline.py`.

## Problem Description

### Symptoms
- When users configured `literal_definitions` in the parser settings dialog, the settings were ignored by the Python parser
- `literal_definitions` always came as the default from YAML (e.g., `[{attr_key: "style", attr_value: "rounded=1"}]`), regardless of user settings
- When the user explicitly cleared all literal definitions (empty array), the defaults were still used

### Expected Behavior
- If user sets custom `literal_definitions`, Python should receive them
- If user clears all definitions (empty array `[]`), Python should receive an empty array
- If no override is provided (`null`), Python should use defaults from `default.yml`

### Root Cause Analysis

The issue was in `aicode/python_core/pyodide_pipeline/drawio_pipeline.py` in the `_apply_parser_overrides` function.

#### Data Flow (TypeScript to Python)

1. **TypeScript Side** (`aicode/typescript_plugin/src/rdfexport.ts`):
   - `buildParserConfigPayloadFromSettings()` creates a `DrawioParserConfigPayload` object
   - Includes `literal_definitions` field via `denormalizeLiteralDefinitions()`
   - Properly converts from internal format to `{attr_key, attr_value}` format

2. **Python Side** (`aicode/python_core/pyodide_pipeline/drawio_pipeline.py`):
   - `parse_drawio_xml()` calls `_apply_parser_overrides(parser_config)`
   - `_apply_parser_overrides()` handles many fields but **WAS MISSING** `literal_definitions`

#### Comparison: Working vs Broken

**Working field (`metacharacter_substitute`):**
```python
if "metacharacter_substitute" in overrides:
    config["metacharacter_substitute"] = _normalise_metacharacters(
        overrides["metacharacter_substitute"],
    )
```

**Broken field (`literal_definitions`):**
```python
# NO HANDLER EXISTED - field was completely ignored!
```

## Solution

### Changes Made

#### File: `aicode/python_core/pyodide_pipeline/drawio_pipeline.py`

**1. Added normalizer function** (after `_normalise_metacharacters`):

```python
def _normalise_literal_definitions(
    value: Any,
) -> list[dict[str, str]] | None:
    """Normalise literal_definitions from overrides.

    Returns:
        None: if value is None (use defaults from YAML)
        []: if value is an empty list (user explicitly cleared definitions)
        list of dicts: if value contains valid entries
    """
    if value is None:
        return None

    if not isinstance(value, list):
        return None

    result: list[dict[str, str]] = []
    for item in value:
        if not isinstance(item, dict):
            continue
        attr_key = item.get("attr_key")
        attr_value = item.get("attr_value")
        if isinstance(attr_key, str) and isinstance(attr_value, str):
            result.append({"attr_key": attr_key, "attr_value": attr_value})

    # Return empty list if input was empty list (user explicitly cleared)
    # This is different from None which means "use defaults"
    if len(value) == 0:
        return []

    return result
```

**2. Added handler in `_apply_parser_overrides`** (after `rml_enabled` handler):

```python
if "literal_definitions" in overrides:
    normalised = _normalise_literal_definitions(
        overrides["literal_definitions"]
    )
    # Only override if normalised is not None
    # None means "use defaults from YAML"
    # Empty list means "user explicitly cleared definitions"
    if normalised is not None:
        config["literal_definitions"] = normalised
```

## Technical Notes

### Semantic Difference: `null` vs `[]`

This fix preserves an important semantic distinction:
- `null`: "No preference specified, use defaults"
- `[]`: "User explicitly cleared all definitions"

This matches the behavior of the TypeScript side's `denormalizeLiteralDefinitions()` which:
- Returns `null` if input is not an array (use defaults)
- Returns empty array if input is empty array (user cleared)
- Returns populated array if entries exist

### Consistency with metacharacter_substitute

The fix follows the same pattern as `metacharacter_substitute` handling, maintaining code consistency throughout the override application logic.

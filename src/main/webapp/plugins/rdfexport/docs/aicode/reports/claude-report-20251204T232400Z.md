The feature provides a clean separation between default configuration (defined in TypeScript as the single source of truth) and runtime behavior in the Python parser.

## Task Description

Implement a configurable "literal definitions" feature that:
- Reuses visual elements from previous attempts
- Reads defaults from configuration (single source of truth in TypeScript)
- Exposes literal definitions in the UI
- Reliably serializes user input and passes to cell classifier
- Implements proper logic in `_style_suggests_literal` function

### Requirements
- Default value: `[{attrKey: "style", attrVal: "rounded=1"}]`
- If literal_definitions is None: use defaults from TypeScript config
- If literal_definitions is empty array []: everything is a literal (return True)
- If literal_definitions is non-empty array: use attrKey and "attrVal in value" logic
- No hardcoded configuration outside of TypeScript defaults
- No regressions in existing tests

1. **None case**: Falls back to original behavior (should not occur as TypeScript always provides defaults)
   ```python

           # ... original logic
   ```

2. **Empty array case**: Everything is a literal
   ```python
   if len(self._literal_definitions) == 0:
       return True
   ```

3. **Non-empty array case**: Check if any definition matches
   ```python
   for definition in self._literal_definitions:
       attr_key = definition.get("attrKey", "")
       attr_val = definition.get("attrVal", "")

       if attr_key in style:
           if not attr_val:  # Presence check only
               return True
           if attr_val in style:  # Value match
               return True
   ```

## Design Decisions

### 1. Single Source of Truth
- **Decision**: TypeScript layer holds the default configuration
- **Rationale**: Eliminates duplication and ensures consistency across the stack
- **Implementation**: Python fallback behavior mirrors TypeScript defaults but is only used if TypeScript fails to provide config

### 2. Attribute Matching Strategy
- **Decision**: Simple substring matching for `attrVal in style`
- **Rationale**: Handles various style formats without complex parsing
- **Examples**:
  - `attrKey="style", attrVal="rounded=1"` matches `"style=rounded=1;html=1"`
  - `attrKey="ellipse", attrVal=""` matches any style containing "ellipse"

### 3. UI Design
- **Decision**: Inline add/remove entries similar to metacharacter substitution section
- **Rationale**: Consistent with existing UI patterns, familiar to users
- **Benefits**: Easy to learn, flexible for any number of definitions

### 4. Empty Array Semantics
- **Decision**: Empty array means "everything is a literal"
- **Rationale**: Provides an easy way to override all detection logic
- **Use Case**: Useful for debugging or when all nodes should be treated as literals
